# Rules engine documentation index

This index exists because the severity-scoring documentation grew large
enough that architecture and calibration questions were being discussed
in the same breath, even though they answer different questions and
carry different levels of certainty. Part A is settled, implemented, and
tested. Part B is calibration analysis on top of Part A — real findings,
but corpus-level, single-reviewer, and in several places preliminary.
Read them as separate conversations.

## Part A — Deterministic rule engine (detection, not severity)

How a clause gets found and matched in the first place. This layer is
implemented, tested (261+ tests), and not part of any open calibration
question.

- `rules_engine_overview.md` — how the engine works end to end: chunking,
  pattern/anchor matching, `RuleClass` (presence-risk vs required-section)
- `rule_structure.md` — the `Rule` dataclass fields used for detection
  (pattern, anchors, nearby, window, topic/protective patterns)
- `rule_categories.md`, `all_rules.md`, `example_rules.md` — the rule
  catalog itself, organized by severity tier and practice area
- `versioning_strategy.md` — how `rules/version.json` and rule-count
  changelogs are maintained

## Part B — Severity calibration (WAS, ceilings, thresholds)

Given a matched clause, how severity gets assigned. The scoring *engine*
(`severity_scoring.py`) is implemented and tested, now at **Framework
v1.1.0**. The 11 factors and 8 ceiling rules are still frozen per the
architecture doc below; v1.1 changed only the non-ceiling threshold
mechanism, via a governed process, not a silent edit.

**Read this first:**
- `severity_v1_1_release_notes.md` — the current, authoritative state:
  what v1.1 changed, why, the evidence, and what's still not done
  (independent second-reviewer sign-off, wiring into the live engine).

**Frozen architecture (factors and ceiling rules — approved v1.0, not
redesigned by v1.1):**
- `severity_architecture.md` — the v1.0 spec: 11 factors, 8 ceiling
  rules, WAS formula, original absolute thresholds, metadata schema,
  jurisdiction model
- `severity_implementation.md` — how that spec was turned into code:
  engine, validators, migration tooling, contributor workflow

**Migration (applying the frozen factors/ceilings to the 117 real
rules):**
- `severity_migration_report_full_117.md` — every rule's legacy vs.
  computed severity **under the current v1.1 default**, generated by
  `scripts/generate_full_migration_report.py` from
  `scripts/severity_migration_full.py`. Still a single-reviewer pass
  ("Reviewer 1 (Framework Author) Migration") — the blind re-score gate
  and attorney sign-off have not happened yet, regardless of which
  threshold mode is default.
- `severity_migration_report_full_117_v1_0_absolute_archive.md` — the
  pre-v1.1 snapshot under the original absolute mode, preserved as the
  calibration record that motivated the change.

**Calibration experiments and the resulting fix (in the order they were
produced — each one responds to what the previous one found):**
1. `severity_practical_max_experiment.md` — per-rule ceilings given each
   rule's own already-scored factors
2. `severity_family_clustering_experiment.md` — same question, grouped
   by practice-area family; the "no single threshold fits every family"
   finding
3. `severity_family_conclusion_stress_test.md` — an adversarial attempt
   to disprove #2 by testing five alternative explanations
4. `severity_authoring_bias_and_confidence.md` — whether the family
   pattern is actually an authoring artifact, plus a per-family
   confidence table by sample size
5. `severity_threshold_v1_1_candidate.md` — the fix proposal responding
   to #2 (superseded by the release notes above, kept for its detailed
   rationale and evidence table)

### How much weight to put on Part B's findings right now

- The scoring *engine* is solid: 391 tests, deterministic, matches its
  own spec exactly in both the current default (relative) and the
  preserved compatibility mode (absolute).
- The threshold mechanism **has** now changed, deliberately and through
  a recorded process (evidence gathered, alternatives tested, explicit
  sign-off, full regression re-run, old default preserved as a named
  mode) — not a silent edit to the frozen factors or ceiling rules,
  which remain exactly as approved in v1.0.
- The calibration *findings* underlying that change should still be read
  at the confidence level stated in `severity_authoring_bias_and_confidence.md`
  — 8 of 18 families have n≤4 rules and are explicitly preliminary.
- None of Part B has been through the §10 blind re-score gate (a second,
  independent reviewer) — this is true of both the v1.0 and v1.1
  numbers, since v1.1 changed how existing single-reviewer vectors are
  interpreted, not who scored them. Treat every calibration conclusion
  as a hypothesis for that review, not a final answer.
- **Not yet done, regardless of v1.1**: wiring any of this into
  `rules_engine.py`'s live `Rule.severity` field. This framework still
  runs in parallel to production, exactly as before this release.
